<acceptance_criteria>

## AC-1: Thread safety sur current_status
```gherkin
Given le motor_service tourne avec ContinuousHandler actif
When le daemon thread et le main thread accèdent à current_status simultanément
Then toutes les lectures/écritures sont protégées par un threading.Lock
```

## AC-2: Validation des commandes GOTO/JOG
```gherkin
Given une commande GOTO/JOG arrive via IPC
When l'angle est NaN, négatif, >360, ou la vitesse invalide
Then la commande est rejetée avec un message d'erreur clair dans le status
  And aucun mouvement moteur n'est déclenché
```

## AC-3: ContinuousHandler thread join fiable
```gherkin
Given un ContinuousHandler avec un thread actif
When stop() est appelé et thread.join(timeout=2.0) expire
Then un warning est loggé indiquant que le thread est toujours actif
  And le nouveau mouvement ne démarre pas tant que le thread précédent vit
```

## AC-4: IPC locking simplifié
```gherkin
Given le motor_service écrit le status via ipc_manager
When write_status() est appelé
Then fsync() n'est pas appelé sur tmpfs
  And le verrouillage est cohérent (même fichier pour reader/writer)
  And la sémantique rename atomique est préservée
```

## AC-5: Error recovery uniforme
```gherkin
Given une erreur de tracking se produit
When le status passe à 'error'
Then error_timestamp est toujours défini
  And _check_error_recovery() peut déclencher la récupération automatique
```

## AC-6: Pas de régression
```gherkin
Given les modifications appliquées
When on exécute uv run pytest
Then tous les tests existants passent (≥ 406 tests verts)
```

  <name>Task 1: Thread safety et validation dans command_handlers.py</name>
  <files>services/command_handlers.py</files>
  <action>
    **[S-01/S-07] Thread safety current_status:**
    - Ajouter un paramètre `status_lock: threading.Lock` au constructeur de chaque handler
    - Dans ContinuousHandler._movement_loop(), wrapper chaque modification de current_status avec `with self.status_lock:`
    - Le lock est créé dans motor_service.py et passé aux handlers

    **[S-09] Validation des commandes:**
    - Créer une fonction `_validate_angle(angle) -> tuple[bool, str]` en haut du module
    - Vérifie : isinstance(angle, (int, float)), math.isfinite(angle), 0 ≤ angle ≤ 360
    - Appeler dans GotoHandler.execute() et JogHandler.execute() avant tout mouvement
    - En cas d'échec : logger l'erreur, mettre current_status['status'] = 'error' avec message, return sans bouger

    **[S-08] Thread join fiable:**
    - Dans ContinuousHandler.stop() : après thread.join(timeout=2.0), vérifier thread.is_alive()
    - Si alive: logger warning "ContinuousHandler thread did not stop within timeout"
    - Dans ContinuousHandler.start() : vérifier qu'aucun thread précédent n'est alive avant d'en créer un nouveau
    - Si un thread est alive : logger error et return sans démarrer

    **Éviter:**
    - Ne PAS changer la logique métier des handlers (mouvements, calculs)
    - Ne PAS toucher à TrackingHandler (sera traité en Plan 02 si nécessaire)
    - Ne PAS ajouter de threading pour GOTO/JOG (trop risqué pour un refactoring — à évaluer dans un milestone futur dédié)
