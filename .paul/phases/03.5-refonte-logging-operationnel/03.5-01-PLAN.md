<objective>
## Goal
Refondre le système de logging pour le rendre exploitable en débogage distant : format structuré clé=valeur grep-friendly, heartbeat motor_service, snapshot IPC /dev/shm dans les logs, milestones session tracking, et couverture encodeur.

## Purpose
L'observatoire est sur le terrain, le développement est à distance. Les logs actuels couvrent 25% des opérations, sont verbeux en texte libre avec emojis, et ne prouvent jamais que le système est vivant. L'incident LBN (12h de trou dans les logs) a démontré que le diagnostic distant est aveugle.

## Output
- Format structuré `action | clé=valeur` sur tous les messages opérationnels
- Heartbeat motor_service toutes les 10s
- Snapshot IPC /dev/shm dans les logs toutes les 60s pendant tracking
- Milestones session tracking toutes les 5 min
- Health encodeur périodique
- Tests validant le format et la présence des logs critiques
</objective>

```gherkin

And le heartbeat est visible même quand aucune commande n'est reçue
```

## AC-2: Snapshot IPC dans les logs
```gherkin
Given le tracking est actif

Then un log INFO est émis avec format "ipc_snapshot | status=... position=... mode=... encoder_angle=... encoder_calibrated=..."

```

## AC-3: Session milestone toutes les 5 min
```gherkin

```

    **Heartbeat dans la boucle principale de MotorService.run()**

      ```
      logger.info(f"heartbeat | uptime={uptime} tracking={is_active} object={obj} corrections={n} encoder={status} cmds={cmd_count}")
      ```

    - Lire current_status (déjà disponible en mémoire)

      ```
      logger.info(f"ipc_snapshot | status={status} position={pos:.1f} target={target} mode={mode} encoder_angle={enc:.1f} encoder_calibrated={cal} tracking_object={obj}")
      ```
    - Ceci rend le contenu de /dev/shm consultable dans les logs fichier

    **Logging des commandes IPC**

    Dans `process_command()`, après la ligne `logger.info(f"Traitement commande: {cmd_type}")`, remplacer par :
